## Why

`substitutions.toml` and the `propose_substitutions` tool encode a deterministic substitution engine that, in real use, is **never curated**. Onboarding (the just-shipped `improve-onboarding` change) deliberately doesn't prompt for it; the file is designed to accrete reactively, and in practice it sits empty. So the capability is dead weight: a file (shared + per-tenant override), a rule engine (`src/substitutions.ts`), a tool, a spec, a test suite, a `kroger_flyer` flag, and five threads of agent persona — all dormant until someone hand-writes TOML nobody writes.

The determinism it guards is also thinner than it looks. The substitution *decision* was never deterministic — the spec itself says the tool "surfaces alternatives for the agent to present to the user for confirmation" and "SHALL NOT itself apply a substitution." The agent + user always made the call; the rule file only deterministically *enumerated candidates*. That enumeration is exactly what an LLM's world knowledge does well, and does **better** than a flat rule list on messy real input (it catches context-sensitive swaps a static `acceptable_substitutes` array can't express).

The file's two genuinely-irreducible halves — **vetoes** ("never tilapia for salmon") and **ranked preference** ("reach for arctic char first") — are personal, not world knowledge. But they already have a home the LLM reads at proposal time: `taste.md` / `preferences.toml [dietary]`. So removal rehomes the irreplaceable bits rather than dropping them.

This change is also the **beachhead** for a restated determinism boundary (see `design.md`): *deterministic mechanisms should guard keys and gates over unloadable/shared data, not make decisions over loadable per-tenant state.* Substitutions is the lowest-risk first application; aliases and the shrinking of `verify` are follow-ons the principle implies, staged after this ships and is validated.

## What Changes

- **Remove the `ingredient-substitution` capability.** Delete `propose_substitutions`, `src/substitutions.ts` (parse / merge / `findRule` / `proposeInventory` / `proposeSale`), and `test/substitutions.test.ts`. Delete the shared and per-tenant `substitutions.toml` and the `docs/data-template/substitutions.toml` stub.
- **Retire the per-tenant override-merge layer.** `mergeSubstitutionRules` (the shared+override join) was substitutions' *sole* consumer; it goes with it. The `shared-corpus` "Shared reference data" requirement drops the substitutions override clause (keeping aliases).
- **Retire `verify_pantry_for_recipe` / `verify_pantry_for_candidates` entirely** (not just the sub bucket). The pantry now enters context via `read_pantry` loaded **up front as a selection input** in `meal-plan`, before recipes are chosen, so the LLM proposes a pantry-aware menu *and* spots stand-ins ("recipe wants salmon, you've got trout") by reasoning over it. The precise to-buy list — `verify`'s real output — moves into the reasoning pass: at cost/confirm the agent loads the chosen recipes' full content (`read_recipe`, which it needs to cook anyway), matches their ingredients against the loaded pantry, and emits `grocery_list_ops` directly. Consumer check confirms no other dependency: `place_order` works off `grocery_list ∪ caller-supplied menu_needs` (it never parsed recipes), and `perishable_ingredients` classification uses its own normalization — so the recipe-ingredient parser (`recipe-ingredients.ts`) falls out with `verify` too. `mark_pantry_verified` stays (its trigger shifts to LLM reasoning over the loaded pantry dates).
- **What this trades:** `verify`'s deterministic, exhaustive bucketing (parse → normalize → exact/fuzzy match → age math) becomes LLM reasoning over the loaded recipe + pantry, confirmed conversationally. The exhaustiveness guarantee is the one real loss; it is low-stakes (a missed item is recoverable and caught by the iterate-before-commit step) and consistent with the thin-tools direction — semantic matching ("scallion ≈ green onion") actually improves, since string-fuzzy never could.
- **Sale-sub and unavailable-fallback substitution become LLM-enumerated Kroger searches.** `kroger_flyer` drops the `against_substitutions` flag (the tool keeps `terms`); the agent enumerates substitute candidates from world knowledge and passes them as explicit `terms` / `kroger_prices` inputs. These were always Kroger-side (enumerate-then-search), so nothing structural changes but the source of the candidate list.
- **Remove the `update_substitutions` write tool** and the `"substitutions"` value from the `commit_changes` `config_updates` enum and the curated-config routing.
- **Rehome the capture path.** When a member voices a standing substitution preference or a veto, the agent offers to save it to `taste.md` (honored prose), not a rule file.
- **Docs.** `ARCHITECTURE.md` (5 refs incl. the stale build-validation claim — the build does not actually validate substitution rules), `SCHEMAS.md` (drop the `substitutions.toml` section), `TOOLS.md` (drop `propose_substitutions` / `update_substitutions` / the flyer flag). Capture the restated boundary principle + follow-up roadmap in `design.md`.

## Capabilities

### Removed Capabilities
- `ingredient-substitution`: the deterministic rule engine and `propose_substitutions` are removed; substitution becomes LLM reasoning (inventory subs over the loaded pantry; sale/unavailable subs as enumerated Kroger searches), with vetoes/preferences captured as `taste.md` prose.

- `pantry-verification` (**Removed**): `verify_pantry_for_recipe` / `verify_pantry_for_candidates` are retired outright; the capability's deterministic parse/match/bucket logic moves into `meal-plan` reasoning over the loaded pantry + recipe content.

### Modified Capabilities
- `menu-generation`: the context pre-pass loads the raw pantry up front as a *selection* input; inventory-substitution spotting and the to-buy list move from `verify` to LLM reasoning over the loaded pantry and the chosen recipes' content (`read_recipe`).
- `kroger-integration`: `kroger_flyer` no longer derives precise terms from `substitutions.toml` (no `against_substitutions`); the caller passes substitute candidates as explicit `terms`.
- `data-write-tools`: removes `update_substitutions`, the `"substitutions"` `config_updates` enum value, and the per-tenant-substitution-override routing clause.
- `shared-corpus`: "Shared reference data" drops the `substitutions.toml` shared-default + per-tenant-override clause (aliases unchanged).
